# Code of Conduct

Short version: be useful, be honest about limits, no harassment, no
asshole behaviour.

## Scope

This applies in every project space — issues, pull requests,
discussions, code review, the project's GitHub Discussions tab, and
any direct outreach about the project (email, social DM).

## Expected behaviour

- **Use the project surface.** Bugs go in Issues, design in
  Discussions, security in email. Crossposting noise is fine the
  first time you don't know; repeated misrouting after the
  maintainer asks you not to is not.

- **Disagree on technical merit.** "Your design has the wrong
  Mechanism 1 invariant because X" is welcome. "Your design is
  stupid" is not.

- **Honour the contribution shape.** If a PR closes an issue, link
  the issue. If a PR builds on a Discussion, link the Discussion.
  This keeps the project navigable for the next person.

- **Be honest about limits.** Don't claim a feature works on
  platforms you have not tested. Don't push a benchmark number you
  have not actually run. mgi-mind is built around honest limits in
  release notes; PR culture follows.

## Unacceptable behaviour

- Harassment, slurs, or targeted personal attacks against any
  contributor regardless of demographic or affiliation.

- Sexualised language or imagery, in any project space.

- Doxxing — publishing private information about a contributor
  without their consent.

- Sustained off-topic disruption — repeatedly redirecting
  Discussions or PR threads to unrelated agendas after being asked
  to stop.

- Sockpuppeting, vote manipulation on GitHub stars / reactions,
  paid-shill posting without disclosure.

## Enforcement

The maintainer (currently @madgodinc) is the enforcement point. If
behaviour above happens, you can:

- **Comment in the thread** asking for the behaviour to stop. Usually
  enough.
- **Mark the issue / PR @madgodinc** if comment did not work.
- **Email** for incidents that should not be public (harassment,
  doxxing, private outreach). Contact in the project's GitHub
  profile.

Possible responses, in escalating order: warning, hide / delete the
specific message, temporary ban from project spaces, permanent ban.
The maintainer decides; this is not a tribunal.

## Out of scope

- Private behaviour between contributors outside project spaces is
  not covered. If it bleeds into the project (someone shows up in
  an Issue thread with off-project grievances), it becomes in
  scope.

- Disagreements about architecture, design choices, or roadmap
  priorities are not Code-of-Conduct issues. They are technical
  disagreements; argue them in Discussions or the relevant PR
  thread.
